# 活动写入配额扫描修复计划

**目标：** 数据接近 100 MiB 上限时，清理不得逐帧全盘扫描或拖慢自动化；正常退出后数据必须保持在硬上限内。

**根因：** `register_write()` 已维护精确容量账本，但超限时调用 `_enforce()` 未传入账本值，导致每个帧文件和事件写入都重新遍历整个数据树。日志线程积压后无法在退出时排空，进程可能停在文件写入与配额登记之间。

## 实施步骤

1. 在 `try/tests/test_storage.py` 新增失败测试：初始化只允许扫描一次数据根目录，连续两次活动写入触发清理时不得再次扫描。
2. 在 `src/auto_fishing/storage/quota.py` 让活动写入超限路径把 `_known_total` 传给 `_enforce()`；不改变清理顺序、保护文件或异常策略。
3. 运行存储专项、完整测试、候选构建和烟雾；在 `try` 副本验证当前 100 MiB 生产布局仅按需删除最旧帧。
4. 更新 `AGENTS.md`、`doc/验收标准.md` 与 `doc/进展记录/2026-7-14.md`，备份并替换根目录 EXE，合并本地 `main`，清理任务分支。
5. 重新审计生产数据并执行一次启动清理，使其回到硬上限内；所有验证通过后推送私有 GitHub。
