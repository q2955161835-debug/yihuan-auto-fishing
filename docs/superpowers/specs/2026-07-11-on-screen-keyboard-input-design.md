# Windows 屏幕键盘输入适配设计

## 背景与目标

真实《异环》验收确认：程序直接调用 `SendInput` 发送 F 时，Windows 返回全部发送成功，但游戏没有响应；相同画面下实体键盘 F 成功。用户随后用实体鼠标点击 Windows 屏幕键盘 F，游戏成功进入钓鱼控制画面。目标是在不安装驱动、不注入游戏进程的前提下，让正式程序通过可见的 Windows 屏幕键盘完成 F、A、D 输入，并保持现有画面识别、F8、安全暂停、完整日志与单文件发布能力。

## 已确认约束

- 自动化运行期间 Windows 屏幕键盘始终可见，不隐藏，固定在游戏所在显示器左下角。
- 程序独占鼠标；用户运行期间不操作鼠标键盘。F8 仍作为紧急暂停入口。
- 屏幕键盘由程序启动或复用。若启动前已经存在，退出时保留；若由本程序启动，退出时在释放全部输入后关闭。
- 继续使用 DXcam 桌面复制截屏，不改为游戏窗口独立捕获。
- 顶部进度条和右下角准备/上钩区域不得被屏幕键盘遮挡；只有结算识别排除左下角屏幕键盘的重叠像素。
- 不使用后台鼠标窗口消息代替可见点击。实测向 `OSKMainClass` 直接发送鼠标消息没有触发游戏输入。

## 架构与组件

新增 `platform/on_screen_keyboard.py`，包含两个职责单一的组件：

1. `OnScreenKeyboardWindow` 管理系统屏幕键盘生命周期与几何信息。它查找 `OSKMainClass` 顶层窗口；不存在时启动 `%WINDIR%\System32\osk.exe` 并在限定时间内轮询窗口。它记录是否由本程序启动，将窗口置顶、调整为可用尺寸并停放在目标显示器左下角。每次输入前重新读取窗口与客户区坐标，防止用户移动、缩放、关闭或切换布局后仍使用陈旧坐标。
2. `OnScreenKeyboardInputBackend` 实现现有 `InputBackend` 协议。它复用 Win32 鼠标输入原语，但键盘事件不再直接发送扫描码：F、A、D 映射为默认标准屏幕键盘客户区中的归一化按键中心，先通过 `ClientToScreen` 换算为屏幕坐标，再移动鼠标并发送鼠标按下/抬起。

现有 `SafeInput` 保持业务语义：`tap_f()` 仍产生平衡的 F 按下/抬起；`set_direction()` 换向时仍先释放旧方向；`release_all()` 仍释放所有逻辑按键。输入后端内部只允许一个屏幕键盘按键处于鼠标按住状态，并维护实际按住键与鼠标状态，确保异常清理不依赖上层集合是否已经成功更新。

`Application` 在启用 DPI 感知后准备屏幕键盘，在关闭阶段按“停止热键 → 关闭引擎 → 释放输入 → 关闭自启动的屏幕键盘 → 关闭日志”顺序清理。绑定游戏窗口或显示器发生变化后，引擎把目标显示器矩形交给输入后端重新定位屏幕键盘。

## 输入行为与坐标

- 按键中心使用相对于屏幕键盘客户区的归一化坐标，不写死 2048×1152 或特定 DPI 的绝对像素。首个实机基线来自 1350×377 客户区的标准布局，并通过单元测试固定 A、D、F 的相对顺序、行位置和边界。
- F 按下：确认屏幕键盘有效且位于目标显示器左下角，移动鼠标到 F 中心并发送左键按下。F 抬起：发送左键抬起。现有 50 毫秒轻点时长不变。
- A/D 按下：先确认没有其他键处于实际按住状态，移动鼠标到目标按键中心并发送左键按下，保持鼠标在该键上。
- A/D 换向：发送左键抬起，确认旧键状态清除，再移动到新键并按下。
- 任意抬起、暂停、F8、窗口失焦、截图异常、识别异常、完成或退出：发送全局鼠标左键抬起，并清空实际按住键。重复释放必须幂等。
- 结算界面的现有游戏坐标点击仍使用普通鼠标轻点；执行前必须先释放屏幕键盘方向键。真实验收若证明游戏拒绝该点击，再根据结算画面可用按键单独扩展，不把未验证按键猜测写入本阶段。

## 屏幕键盘几何与遮挡

屏幕键盘以目标显示器工作区左下角为锚点，保持完整可见，并使用经实机验证的最小可用尺寸。每次定位后读取窗口矩形、客户区矩形与客户区屏幕原点，校验：

- A、D、F 三个按键中心均位于屏幕键盘客户区和目标显示器内。
- 屏幕键盘不与 `TOP_ROI` 顶部进度区域相交。
- 屏幕键盘不与 `READY_ROI` 右下角准备/上钩区域相交。
- 窗口类别仍为 `OSKMainClass`，窗口可见且未最小化。

屏幕键盘矩形从显示器坐标换算到游戏客户区坐标，作为可选遮挡矩形传给 `SceneRecognizer`。进度和准备识别沿用原始像素；结算识别的暗色、蓝色比例只在 `RESULT_ROI` 中未被遮挡的有效像素上计算。若有效像素不足预设下限，或屏幕键盘意外进入顶部/右下角关键区域，自动化以 `E_OSK` 安全暂停，而不是继续使用污染画面。

运行日志与逐帧 JPEG 保存未经掩码的原始客户区画面，便于复核屏幕键盘位置；结构化事件额外记录窗口句柄、所有权、窗口矩形、客户区尺寸、按键、目标坐标、鼠标按下/释放结果和遮挡矩形。

## 故障处理

下列情况统一转换为 `InputFailure`，由引擎按现有输入错误路径以 `E_INPUT` 暂停；能够在输入前识别的屏幕键盘几何/生命周期错误使用更明确的 `E_OSK`：

- 启动超时、窗口不存在、窗口关闭、最小化、窗口类别不匹配。
- 目标显示器过小，无法同时完整显示屏幕键盘且避开顶部和右下角关键识别区域。
- 客户区尺寸非法、按键中心越界、屏幕坐标换算失败。
- `SetCursorPos`、鼠标按下或鼠标抬起没有成功。
- 换向或异常清理时旧键释放失败。

输入后端在任何部分失败后都立即尝试额外发送一次鼠标左键抬起，并把原始错误与清理错误同时写入日志。程序不得因为关闭屏幕键盘失败而跳过输入释放或日志关闭。

## 测试与验收

### 自动测试

- 生命周期：复用已有窗口、自启动、启动超时、只关闭自启动实例、关闭失败不跳过其他清理。
- 几何：不同显示器原点、负坐标、DPI/尺寸、左下角定位、按键归一化换算、顶部和右下角遮挡拒绝。
- 输入：F 平衡轻点、A/D 按住、换向先释放、重复释放、移动失败、按下失败、抬起失败及二次清理。
- 视觉：结算比例排除遮挡像素；顶部进度和右下角准备识别不使用掩码；有效像素不足拒绝识别。
- 引擎与应用：绑定后重新定位、`E_OSK` 暂停、F8/失焦/关闭释放、生命周期清理顺序、结构化日志字段。
- 全量 pytest、PyInstaller 单文件构建与发布物启动/响应/关闭烟雾全部通过。

### 真实游戏验收

1. 程序启动后自动打开或复用屏幕键盘并将其固定在游戏显示器左下角。
2. 点击开始并切回游戏，自动点击屏幕键盘 F 后必须进入等待上钩；日志和截图证明实际命中 F。
3. 上钩后再次自动点击 F，进入进度条控制。
4. 在进度控制中分别观察 A 按住、释放、D 按住、换向和安全带释放，游戏方向与识别决策一致，任意时刻不同时按 A/D。
5. 控制阶段按 F8，鼠标左键立即释放；继续后能重新分类画面并恢复。
6. 完成至少一条鱼：识别结算、关闭结算、回到准备状态并把成功数从 0 更新为 1。
7. 再完成连续 5 条鱼，期间没有屏幕键盘遮挡导致的进度/准备误判，没有鼠标或方向键残留。

只有上述真实游戏单轮和连续 5 轮均通过，才能声明“可成功自动钓鱼”并合并 `main`。
