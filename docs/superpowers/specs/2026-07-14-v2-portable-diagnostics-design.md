# 异环自动钓鱼 V2 免安装与按需诊断设计

## 1. 目标与边界

V2 的目标是把已经通过真实游戏验收的 V1 自动钓鱼核心，发布为可直接分发给其他用户的 Windows 单文件程序：无需 Python、无需额外下载运行库，支持 64 位 Windows 10 和 Windows 11，正常运行不持续写运行日志；发生自动错误或用户主动点击“报告错误”时，才生成一个便于转交排查的诊断压缩包。

本阶段不改动已经验收通过的钓鱼状态机、画面特征、黄标追踪算法、A/D 控制参数、F 输入节奏和结算时序。兼容性工作优先复用 V1 的物理坐标、归一化识别、多显示器映射和屏幕键盘输入实现，只补已确认的缺口，不建立第二套截图、坐标或输入链路。

V2 仍遵守既有安全边界：不读取游戏内存、不注入游戏进程、不规避反作弊，只使用屏幕截图、Windows 系统屏幕键盘和标准鼠标输入；F8 继续作为全局紧急暂停键。

## 2. V1 兼容性复核结论

### 2.1 直接复用的能力

以下能力已经存在且有自动测试，不在 V2 中重复设计：

- 进程在创建 Tk 控制窗口前启用 DPI 感知；发布清单声明 `PerMonitorV2`。
- 游戏客户区通过 `GetClientRect` 与 `ClientToScreen` 换算为屏幕物理坐标，不使用固定桌面坐标。
- DXcam 输出按 Windows 显示器设备名和物理分辨率映射，支持多显卡/多输出索引与负屏幕坐标。
- 游戏客户区必须完整位于一个显示器内；跨显示器窗口会安全拒绝，避免从两个缩放空间拼接错误画面。
- 运行中每 0.5 秒刷新客户区、显示器与输出索引；窗口移动或尺寸变化时重算屏幕键盘位置，跨输出时重启对应截图源。
- 所有视觉区域使用客户区归一化坐标；截图先按客户区相对显示器的物理偏移裁剪，因此窗口、无边框和全屏共用同一识别链路。
- 屏幕键盘每次绑定按固定外框宽高比重建，不继承旧尺寸；每次输入前重新读取窗口和客户区，验证完整键盘布局后再按归一化 F/A/D 坐标点击。
- 屏幕键盘支持负显示器坐标、左下角定位、关键识别区遮挡拒绝和 Windows 错误 5 的安全降级。
- `WDA_EXCLUDEFROMCAPTURE` 失败是可见但非致命警告，兼容不支持该能力的 Windows 10 版本。

本设计复核时，现有几何、窗口、屏幕键盘和打包相关 71 项测试全部通过。

### 2.2 需要补齐的缺口

V2 只新增以下兼容性工作：

1. **低分辨率屏幕键盘尺寸。** V1 在 1280×720 显示器上会把 1365 像素标准宽度裁到整屏 1280 像素，因而遮挡右下角上钩识别区并被现有安全校验拒绝。V2 在显示器宽度不足时，仍保持既有固定宽高比与最小客户区安全阈值，但把键盘宽度限制在右下识别区左边；只有实际读取到的客户区仍满足最小宽高和 `3.2～3.9` 宽高比时才允许输入。按键继续使用同一套客户区归一化坐标。
2. **最终发布物 DPI 校验。** V1 的源清单测试包含 `PerMonitorV2`，但构建后校验器只检查管理员权限。V2 必须同时验证最终 EXE 内嵌清单含管理员权限、`uiAccess=false`、`PerMonitorV2` 以及旧式 `true/pm` 回退声明，防止打包工具覆盖 DPI 配置。
3. **运行时 DPI 回退。** 发布物以清单为主；源码运行或旧环境中，先尝试 `SetProcessDpiAwarenessContext(PER_MONITOR_AWARE_V2)`，再尝试 `SetProcessDpiAwareness(PROCESS_PER_MONITOR_DPI_AWARE)`，最后才使用 `SetProcessDPIAware`。任何“已由清单设置”类返回不得导致启动失败。
4. **控制窗口可见性。** V2 使用独立配置，首次启动位置安全；若之后显示器拓扑改变导致保存位置落到虚拟桌面之外，启动时把控制窗口夹回当前可见工作区，不影响游戏客户区坐标。
5. **测试矩阵。** 补充超宽、4K、低分辨率、混合缩放和最终清单验证。Windows 10 实机/虚拟机未提供时，只能声明接口与自动测试兼容，不能写成“Windows 10 实机已验证”。

## 3. 版本与应用装配

### 3.1 共用核心、独立入口

保留现有自动化、识别、截图、窗口、输入和状态机模块作为共用核心。新增轻量产品配置与 V2 启动入口，由入口决定：

- 窗口标题和版本标识；
- 数据根目录；
- 记录器实现；
- 诊断报告服务；
- PyInstaller 规格和产物名称。

V1 与 V2 不复制自动钓鱼代码。V1 的完整落盘记录器和 V2 的内存诊断记录器实现相同的最小事件/帧记录接口，核心只依赖接口，不判断产品版本。

### 3.2 界面与文件名

- V2 窗口标题明确显示“异环自动钓鱼 V2”。
- V2 本地及 GitHub 发布资产名为 `异环自动钓鱼V2.exe`。
- GitHub V1 发布资产名为 `异环自动钓鱼V1.exe`，发布标题和说明明确标记 V1“完整运行日志版”。为保存用户真实验收过的二进制，V1 历史 EXE 内部标题不重建、不篡改；V1 身份由版本标签、发布标题、资产名和哈希确定。
- V2 控制窗在现有按钮区新增“报告错误”，并在诊断成功后显示压缩包完整路径与“打开文件位置”按钮。

## 4. V2 正常运行数据策略

V2 正常运行不创建 `runs` 目录，不写 `events.jsonl`，不持续保存帧截图，也不启用 V1 的 100 MiB 运行日志配额和清理线程。正常运行所需的结构化事件与低分辨率帧只保存在进程内存的固定时长环形缓冲区，关闭程序即消失。

V2 仅允许以下持久化内容：

- 设置：`%LOCALAPPDATA%\异环自动钓鱼V2\config.json`；
- 按需诊断：`%LOCALAPPDATA%\异环自动钓鱼V2\diagnostics\*.zip`。

选择 `%LOCALAPPDATA%` 是 V2 “下载即用”的明确例外：其他用户不一定有 D 盘或 `D:\29551`，而 Windows 10/11 都提供当前用户本地应用数据目录。它通常位于 C 盘，程序界面必须展示实际诊断路径。不得读取或迁移 V1 的 `D:\29551\异环自动钓鱼数据\`，避免两个版本互相影响。

设置文件只含钓鱼数量、控制窗口位置和自动切回游戏开关，不含密钥、用户输入、进程列表或其他敏感信息。

## 5. 内存诊断与压缩包

### 5.1 环形缓冲区

V2 启动后创建进程内存诊断记录器：

- 保存最近 10 秒的结构化事件；事件按核心实际产生速率全部保留，并以时间戳和单调序号排序。
- 以最高 10 FPS 从已处理的最新客户区帧取样，压缩为最长边 480 像素、JPEG 质量 50 后存入内存。
- 缓冲区按时间淘汰，不按磁盘容量清理；记录失败只能丢弃诊断样本并留下内存计数，绝不能暂停或拖慢自动化主循环。

### 5.2 自动错误报告

任何非用户主动暂停的错误状态，包括截图、陈旧帧、视觉、窗口、屏幕键盘、输入、进度丢失和启动/关闭异常，均触发一次诊断生成。流程顺序固定为：

1. 沿用现有安全暂停，先抬起鼠标并释放 A/D；
2. 冻结最近 10 秒环形缓冲区快照；
3. 把错误时刻的最新客户区帧另存为最长边 1280、JPEG 质量 75 的 `error.jpg`；没有可用帧时在元数据中明确记录“截图不可用”；
4. 在后台生成压缩包；
5. 主线程更新界面，显示成功路径或生成失败原因。

同一暂停事件只自动生成一个压缩包；用户继续后发生新的错误可再次生成。诊断生成或清理失败不得改变自动化错误码，不得产生第二次暂停，也不得阻塞退出。

### 5.3 主动“报告错误”

点击“报告错误”后，程序先取消倒计时/恢复令牌，安全暂停并释放所有输入，再以 `manual_report` 作为原因生成同格式压缩包。即使当前未绑定、已经暂停或没有客户区帧，也要生成包含环境、状态和已有事件的报告，并在元数据中说明缺少哪些画面证据。

### 5.4 压缩包内容与保留

每个 ZIP 至少包含：

- `error.json`：V2 版本、报告类型、错误码/文本、当前状态、生成时间、Windows 版本、进程位数、DPI 感知结果、显示器矩形、游戏客户区矩形、屏幕键盘窗口/客户区矩形、截图可用性和诊断丢弃计数；
- `events.jsonl`：最近 10 秒结构化事件；
- `frames/`：最近 10 秒的低分辨率 JPEG；
- `error.jpg`：错误时刻较清晰客户区截图，仅在有可用帧时存在。

不得收集环境变量、令牌、Cookie、完整进程列表、剪贴板、键盘文本或工作区文件。窗口标题默认不写入报告；只记录程序所需的窗口句柄和几何。

压缩包先在诊断目录中使用临时文件生成，完成后原子改名；失败时尽力删除临时文件。目录最多保留最新 5 个完整 ZIP，清理只枚举该目录内符合 V2 命名规则的文件，不扫描其他目录。清理在后台执行，失败只显示警告，不影响主功能。

“打开文件位置”使用 Windows 资源管理器选中具体 ZIP；若文件已被用户移动或删除，界面提示文件不存在，不抛出未处理异常。

## 6. 分辨率、缩放与系统兼容设计

### 6.1 坐标原则

程序内部继续以 DPI 感知后的屏幕物理像素表示显示器、游戏客户区、屏幕键盘和鼠标位置；视觉区域继续以客户区归一化比例表示。V2 不引入逻辑像素与物理像素的手工缩放系数，避免二次缩放。

### 6.2 支持与拒绝边界

- 目标系统：64 位 Windows 10 与 Windows 11。
- 自动测试缩放矩阵：100%、125%、150%、175%、200%。
- 自动测试客户区矩阵：1280×720、1600×900、1920×1080、2560×1440、3440×1440、3840×2160。
- 覆盖单显示器、多显示器、左侧负坐标显示器、不同显示器混合缩放、窗口移动后刷新、窗口/无边框/全屏共用客户区裁剪。
- 游戏客户区跨越两个显示器、低于 960×540、最小化或无法唯一映射 DXcam 输出时继续安全拒绝，不以猜测坐标运行。
- 真正独占全屏下控制窗不保证可见；自动化与 F8 仍按既有设计工作。建议需要观察控制窗时使用无边框或窗口模式。

### 6.3 屏幕键盘兼容策略

宽屏显示器继续使用 V1 已验收的目标：显示器宽度、1365 像素和游戏客户区宽度三分之二之间的安全值。低分辨率显示器则根据右下 `READY_ROI` 左边可用宽度计算较小目标，但仍必须满足：

- 外框保持 `1365:415` 固定比例；
- 实际客户区宽度至少 900、高度至少 250、宽高比为 `3.2～3.9`；
- 整个窗口位于目标显示器内；
- 不与顶部进度区或右下准备/上钩区相交；
- 绑定后和每次点击前都重新读取并验证几何。

若 Windows 10/11 的屏幕键盘主题、布局或响应式行为使这些条件不成立，程序必须以 `E_OSK` 安全暂停并生成诊断，不得猜测按键、不得退回扫描码。

## 7. 错误处理与线程边界

- 自动化线程只负责安全暂停和提交不可变诊断快照，不在主循环中编码整组图片或写 ZIP。
- 诊断工作线程与 Tk 主线程通过现有主线程队列交付结果；任何 Tk 控件更新都在主线程执行。
- 主动报告与自动报告共享同一生成器、保留规则和打开位置逻辑，禁止复制两套打包代码。
- 关闭程序时先释放输入并停止自动化；诊断线程在短超时内尽力完成，超时则安全退出并保留已完成的 ZIP。不得因为诊断线程卡住而重现 V1 日志阻塞主功能的问题。
- V2 没有持续落盘记录器，因此不产生 `E_LOGGING`。内存诊断降级不属于自动化错误。

## 8. 测试与验收

### 8.1 自动测试

实施必须测试先行，至少新增或扩充以下证据：

1. V2 装配不创建 `runs`、不实例化磁盘运行日志；V1 装配仍保持原行为。
2. V2 标题、版本、产物名和独立 `%LOCALAPPDATA%\异环自动钓鱼V2` 路径正确。
3. 环形缓冲区只保留最近 10 秒，截图最高 10 FPS、最长边 480、质量 50；满载或编码失败不影响自动化。
4. 自动错误先释放输入、同一错误只报告一次；继续后新错误可再报告。
5. “报告错误”在运行、暂停、未绑定和无帧状态都能安全生成报告。
6. ZIP 内容、临时文件清理、最多 5 份、目录边界、Unicode 路径与“打开文件位置”参数正确。
7. 上述分辨率与缩放矩阵中，归一化识别区、客户区裁剪、负坐标、多显示器刷新和安全点击坐标保持正确。
8. 1280×720 下屏幕键盘为右下识别区留空且仍满足布局阈值；无法同时满足时明确拒绝。
9. 源清单和最终 V2 EXE 内嵌清单同时包含管理员权限、`uiAccess=false`、`PerMonitorV2` 和旧式 DPI 回退。
10. 全套 pytest、真实 412 帧控制回放、单文件构建、哈希、启动/响应/关闭烟雾全部通过。

### 8.2 人工验收

发布前在当前 Windows 11 实机至少完成：

- V2 首次启动、UAC、窗口标题和正常运行无日志目录；
- 当前游戏分辨率下绑定、屏幕键盘 F/A/D、完整多轮；
- 主动报告错误、自动错误、打开文件位置和第 6 次报告淘汰最旧包；
- 窗口、无边框和可用的全屏模式；
- 125% 或其他非 100% 缩放若当前环境可切换，则至少完成一次绑定和坐标验证。

Windows 10 若没有可用实机或虚拟机，只能记录“代码路径与自动测试通过，实机未执行”。发布说明必须保留这一限制；获得 Windows 10 环境后再补充真实启动、系统屏幕键盘 F/A/D 和诊断包验收。

1. 以当前已验收并推送的 V1 源提交 `6602974` 和根目录 EXE（SHA256 `9A4768D6930F939C32EE8AB23DABA3BA4B98B03351ECCEE9A7ECD96225E52CA4`）创建 GitHub Release `v1.0.0`，资产名为 `异环自动钓鱼V1.exe`。上传后重新下载或读取远端资产信息，核对大小与哈希。
2. 替换本地发布物前，把 V1 EXE 备份到 `D:\0文件夹\备份` 的带时间子目录；同类备份最多保留 2 份。
3. V2 全部自动测试、清单验证、烟雾和 Windows 11 人工验收通过后，合并任务分支到 `main`，删除任务分支，推送私有 GitHub。
4. 在最终 V2 合并提交创建 GitHub Release `v2.0.0`，上传 `异环自动钓鱼V2.exe`，核对远端资产哈希。
5. 只有 V1 远端留存和本地备份均验证成功，才删除根目录旧 `异环自动钓鱼.exe`；根目录保留 `异环自动钓鱼V2.exe` 作为当前版本。

V2 是 PyInstaller 64 位单文件窗口程序，包含 Python 和所需依赖。其他用户仍会看到 UAC 提升提示；由于当前没有代码签名证书，Windows SmartScreen 可能显示未知发布者警告。这两项不应被描述为程序缺少依赖。
