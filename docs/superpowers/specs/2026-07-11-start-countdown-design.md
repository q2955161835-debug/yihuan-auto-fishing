# “开始”倒计时与前台确认设计

## 背景与根因

用户在完成游戏绑定后，需要点击置顶控制窗口中的“开始”。这一点击会让控制窗口成为 Windows 前台窗口。当前实现随后调用 `SetForegroundWindow` 尝试把焦点切回游戏，但 Windows 前台锁策略允许系统拒绝该请求，因此程序显示“无法激活并确认已绑定的游戏窗口”。

绑定本身、游戏窗口句柄和 DXcam 截图均已确认有效。问题发生在控制窗口与游戏窗口之间的前台所有权交接。

## 目标

- 点击“开始”后提供 3 秒倒计时，让用户手动切回已绑定游戏。
- 倒计时结束时只验证游戏是否已处于前台，不再依赖强制抢占焦点。
- 前台验证成功后才启动截屏、状态机和输入线程。
- 验证失败时保持未运行状态，显示可直接执行的中文提示，并允许再次点击“开始”。
- 关闭程序、重新绑定或其他生命周期取消必须使尚未完成的开始倒计时失效。

## 非目标

- 不使用线程输入附加、模拟 Alt 键或其他绕过 Windows 前台锁的方式。
- 不提升管理员权限，不注入游戏进程，不修改现有反作弊边界。
- 不改变绑定倒计时、钓鱼状态机、画面识别阈值和 F8 紧急暂停规则。

## 用户交互

1. 用户完成游戏绑定并设置目标次数。
2. 用户点击“开始”。
3. 控制窗口显示“开始倒计时：3”，随后显示 2、1；开始、绑定、重新绑定和次数输入在此期间不可重复操作。
4. 用户在倒计时内点击已绑定的游戏画面。
5. 倒计时结束：
   - 若已绑定游戏是前台窗口，程序立即开始本轮自动钓鱼。
   - 若不是前台窗口，程序不创建自动化 worker，显示“请在倒计时结束前切回已绑定的游戏窗口”，恢复可操作状态。
6. 用户可以重新点击“开始”再试一次。

## 组件与职责

### `MainWindow`

- 负责目标次数校验、开始倒计时文本展示和控件禁用状态。
- 开始倒计时与绑定倒计时共用“倒计时活动中”门控，避免并发绑定或重复开始。
- 通过完成回调清除倒计时状态并显示同步启动错误。

### `AppController`

- 新增受现有调度器、关闭状态和命令计数保护的延迟开始入口。
- 按 3、2、1、0 调度；到 0 时调用 `engine.start(target)`。
- 使用现有倒计时代际取消机制，确保关闭或重新绑定后迟到回调不再启动引擎。

### `AutomationEngine`

- `start()` 不再主动调用 `activate()` 强制切换前台。
- 在修改核心状态、启动截屏或创建 worker 前调用 `is_foreground(bound)`。
- 验证失败直接抛出明确错误，保持引擎未运行且不产生诊断截图。
- 暂停后的“继续”仍保留既有主动激活与复核逻辑，不在本次范围内改变。

## 数据流与并发

`开始按钮 → UI 校验 → Controller 创建倒计时代际 → 3/2/1 回调 → 用户切回游戏 → Controller 调用 Engine.start → Engine 前台复核 → 自动化启动`

- 同一时刻最多存在一个绑定或开始倒计时。
- 迟到的调度回调必须先核对代际和关闭状态。
- 倒计时期间关闭程序会使代际失效；任何后续回调不得调用引擎。
- 前台复核和引擎启动之间仍使用现有生命周期锁复核绑定及启动状态，避免窗口重绑竞态。

## 错误处理

- 倒计时结束时游戏不在前台：显示“请在倒计时结束前切回已绑定的游戏窗口”。
- 绑定窗口已失效：沿用窗口校验的具体错误并恢复可操作状态。
- 引擎已运行或正在关闭：沿用现有生命周期错误。
- 所有同步错误均通过控制窗口“最近错误”展示，不启动输入或截图线程。

## 测试设计

- UI：点击开始后不立即调用引擎，依次显示 3、2、1；倒计时期间控件锁定。
- Controller：到 0 时只调用一次 `engine.start(target)`。
- Controller：关闭或取消后，迟到回调不能启动引擎。
- Engine：绑定游戏已在前台时启动成功，且不调用 `activate()`。
- Engine：游戏不在前台时同步失败、线程未创建、状态保持可重试。
- 回归：运行期失焦仍进入 `E_WINDOW` 安全暂停；暂停恢复仍主动激活游戏。
- 全量：运行项目全部 pytest 测试、单文件构建和 EXE 启动/关闭烟雾测试。

## 验收标准

- 实机点击“开始”后能看到 3 秒倒计时，并可在倒计时内切回游戏。
- 倒计时结束时游戏在前台，程序进入等待上钩且发送第一次 F。
- 未切回游戏时不启动、不按键，并显示明确提示；再次尝试可成功启动。
- 不引入强制前台抢占、管理员权限、注入或临时固定延时之外的输入行为。
