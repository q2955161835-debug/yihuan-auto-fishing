# 真实鱼获卡片确认与随机重试关闭设计

## 背景与证据

真实运行 `run-20260713T055318102726Z` 已确认自适应窄绿区识别和 A/D 中心控制成功，但结算关闭失败：

- 第 438 帧由控制进度条进入等待结算。
- 第 442 帧由等待结算进入关闭结算，并立即向屏幕坐标 `(384, 792)` 发送一次鼠标点击；日志记录 `requested=2, sent=2`。
- 第 442～443 帧画面仍是紫色漩涡，鱼获卡片尚未出现；当前中央蓝/白比例规则把漩涡连续三帧误判为结算。
- 约第 500 帧起真实鱼获卡片出现，第 600～670 帧稳定可见，但状态机进入关闭结算后不再重试点击。
- 本轮屏幕键盘窗口为 `(0, 921)-(1706, 1440)`，首次点击的 `y=792` 不在键盘内。因此 Windows 接受了点击，但点击发生得过早；现有证据不足以判定游戏是否还会过滤在正确时机发出的模拟鼠标。

低分辨率运行帧量化显示，真实鱼获卡片顶部奖励栏区域具有稳定的白色文字、暗色底和品红色条；紫色漩涡的同一区域没有白色文字。全分辨率超时诊断帧中，该区域白色比例约 4.2%、暗色比例约 53.5%、品红色比例约 14.7%。

## 目标与边界

- 紫色漩涡和过渡黑帧不得触发结算点击。
- 真实鱼获卡片出现后加入受限随机反应延迟，并在卡片仍存在时最多重试三次。
- 每次点击前重新确认卡片仍存在；画面已消失或进入就绪时不得补发迟到点击。
- 点击位置避开左下屏幕键盘与右上控制窗口。
- Windows 报告发送成功不等于游戏已响应；以卡片消失和就绪画面出现作为成功证据。
- 本阶段不更换 Win32 鼠标输入后端、不引入机器学习模型，也不修改进度控制、稳定结束切换和 F/A/D 输入。

## 真实卡片联合识别

保留现有中央结算区域条件，并新增顶部奖励栏区域：

```text
RESULT_HEADER_ROI = (0.38, 0.04) - (0.62, 0.13)
```

顶部奖励栏候选同时满足：

```text
magenta_ratio > 0.08
white_ratio > 0.008
0.30 <= dark_ratio <= 0.80
```

- 品红色范围使用 HSV 色相 135～175、饱和度至少 80、亮度至少 80。
- 白色和暗色沿用现有辅助函数定义。
- 顶部区域同样应用屏幕键盘遮挡有效像素掩码；有效像素不足时沿用现有安全错误。
- `result_candidate` 必须同时通过原中央卡片条件与顶部奖励栏条件。
- `result` 仍要求连续三帧候选，不用固定等待替代视觉确认。

回归夹具使用本轮保存的 480×270 低分辨率帧：第 442 帧作为紫色漩涡反例，第 600 帧作为真实卡片正例；全分辨率 `20260713T055351845668Z_E_TIMEOUT.png` 用于确认正例阈值在生产分辨率下仍有余量。

## 随机延迟与重试状态流

进入关闭结算后不立即点击：

1. 真实卡片连续三帧确认后，采样首次反应延迟 `0.18～0.42` 秒。
2. 到达计划时间且当前帧仍确认真实卡片时，发送第 1 次点击。
3. 点击后若检测到就绪画面，立即完成本轮并停止一切重试。
4. 点击后若卡片暂时消失但就绪尚未出现，保持等待，不发送点击。
5. 只有卡片再次连续确认且已到下一计划时间，才发送重试；每次重试重新采样 `0.40～0.80` 秒。
6. 总点击次数最多 3 次。第 3 次后卡片仍连续存在，则释放输入并以 `E_RESULT_DISMISS` 暂停；既有 8 秒关闭结算超时保留为兜底。

随机采样器作为 `AutomationCore` 的可注入依赖，生产默认使用均匀随机，测试使用确定值；不得通过真实睡眠阻塞 30 帧工作线程，而是使用单调时钟计划下一次点击。

## 点击点与遮挡

首选点击点为游戏客户区归一化坐标 `(0.80, 0.55)`，位于右侧中部，可避开当前左下屏幕键盘和右上控制窗口。发送前读取屏幕键盘实际矩形：

- 若首选点不在屏幕键盘矩形内，使用首选点。
- 若被键盘覆盖，依次尝试 `(0.85, 0.45)`、`(0.70, 0.35)`。
- 三个点均被覆盖时以 `E_OSK` 安全暂停，不得点击屏幕键盘或退回旧的左侧 `(0.15, 0.55)` 坐标。

坐标按当前游戏客户区换算为屏幕坐标；窗口、无边框和全屏模式使用同一归一化规则。

## 日志与错误处理

新增结构化事件：

- `result.dismiss_scheduled`：尝试序号、随机延迟和计划单调时间。
- `result.dismiss_attempt`：尝试序号、屏幕坐标、当前卡片状态。
- `result.dismiss_waiting`：卡片已消失但就绪尚未确认时仅记录状态变化，不逐帧刷日志。
- `result.dismiss_failed`：三次点击均已由 Windows 接受但卡片仍存在，记录尝试数并触发 `E_RESULT_DISMISS`。

现有 `input.request`、`cursor.result` 和 `sendinput.result` 继续记录底层请求与 Windows 返回值。若任意鼠标发送失败，沿用输入异常路径立即释放并暂停，不继续重试。

## 测试与验收

### 自动测试

- 紫色漩涡夹具连续回放不得产生 `result`。
- 真实卡片夹具连续三帧后必须产生 `result`。
- 现有合成结算、夜景拒绝和屏幕键盘遮挡测试继续通过。
- 首次确认后未到随机计划时间不得点击。
- 每次计划时间到达时只有卡片仍存在才点击。
- 卡片消失后不得重试；就绪出现后只计数一次。
- 三次重试的延迟边界、坐标避障、日志字段和 `E_RESULT_DISMISS` 均有确定性测试。
- 全量场景、引擎、状态机、输入和构建测试通过。

### 真实验收

真实单轮必须依次证明：漩涡期间零点击 → 鱼获卡片出现后发送点击 → 卡片关闭 → 就绪识别 → 完成计数从 0 变为 1。若正确时机的三次点击均记录 Windows 成功但卡片不关闭，才确认“游戏疑似过滤直接模拟鼠标”，另行设计间接鼠标输入；本阶段不提前引入该复杂度。
